# 测试工作流

本文件是项目级测试规范，供 AI 和开发者在新增功能、修改行为、修复 Bug、重构组件或调整接口逻辑时使用。本文件维护详细流程、测试文件位置、例外场景和验证命令。

## 基本原则

- 任何影响代码行为的任务，都必须在实施计划中说明测试策略。
- 能通过自动化测试验证的逻辑，应优先补充自动化测试；不能自动化时，必须说明原因和人工/浏览器验证方式。
- 新增功能、行为变化和 Bug 修复，应优先写能描述目标行为的测试，再实现或修改代码。
- Bug 修复必须先复现问题并定位根因，再补充防回归测试，最后做最小修复。
- 测试应验证真实业务行为，不应只验证 mock、stub 或测试工具本身。
- 纯样式、纯文案、纯配置、纯文档变更可以不写自动化测试，但必须说明验证方式。

## 工作流插入点

### 需求与方案阶段

在需求澄清和方案设计时，先识别本次变更的可验证行为：

- 用户可见行为：页面展示、交互、跳转、提示、禁用、loading、empty、error。
- 数据行为：请求参数、响应解析、字段转换、状态存储、缓存和清理。
- 边界场景：接口失败、字段缺失、重复提交、未登录、无权限、空数据。
- 非自动化验证项：视觉布局、遮挡、溢出、浏览器控制台、真实网络请求。

### 计划阶段

`writing-plans` 或其他实施计划中必须包含 `Test Strategy` 或等价章节，写清楚：

- 本次需要新增或修改哪些测试文件。
- 每个测试文件验证什么行为。
- 哪些场景不写自动化测试，以及原因。
- 目标测试命令、全量测试命令和必要的浏览器运行时验证。

计划中的任务如果包含代码行为变更，应把测试作为实现步骤的一部分，而不是完成后的可选补充。

### 执行阶段

- 新增行为：先写描述目标行为的测试，再写实现。
- 修改行为：先调整或新增能体现新规则的测试，再改实现。
- Bug 修复：先写能复现问题的防回归测试，再修复根因。
- 既有稳定行为补测：可以直接补充通过的测试，但需要在回复中说明这是补测既有行为。

## 测试类型与适用场景

| 场景 | 测试类型 | 默认位置 |
| --- | --- | --- |
| 工具函数、常量、字段转换、数据格式化 | 单元测试 | 与源文件同目录，`xxx.test.ts` |
| API 封装、请求参数、响应解析 | 单元测试 | 与源文件同目录，`src/api/xxx.test.ts` |
| Pinia Store、全局状态、持久化读写 | 单元测试 | 与源文件同目录，`src/store/xxx.test.ts` |
| 指令、插件、拦截器等独立机制 | 单元测试 | 与源文件同目录，`src/directives/xxx.test.ts` 或 `src/plugins/xxx.test.ts` |
| Vue 组件展示、props、emit、插槽、基础交互 | 组件测试 | 与组件同目录，`ComponentName.test.ts` |
| 页面内表单校验、提交、loading、error、empty | 组件测试或组件集成测试 | 单页面优先同目录；跨 Store/API/Router 放 `tests/integration/` |
| 路由守卫、登录跳转、跨模块状态流转 | 集成测试 | `tests/integration/` |
| 登录闭环、核心业务主链路、跨页面操作 | E2E 测试 | 先询问用户并说明原因，确认后放 `tests/e2e/` |
| 纯样式、布局、响应式、视觉细节 | 浏览器运行时验证 | 不强制测试文件 |

## 防回归测试规则

防回归测试用于 Bug 修复场景，不是一种独立的测试类型。

- 优先把防回归用例追加到出问题模块已有测试文件中。
- 如果没有对应测试文件，则根据问题影响范围按单元测试、组件测试、集成测试或 E2E 测试新建文件。
- 单模块逻辑问题优先放在源文件同目录的 `xxx.test.ts`。
- 跨 Store、Router、API、页面状态流转的问题放在 `tests/integration/`。
- 只有核心用户链路或跨页面真实操作问题才放在 `tests/e2e/`。
- 防回归用例名称应描述曾经出错的业务行为，例如“登录失败后应刷新验证码并清空验证码输入”。

## 测试文件存放规则

### 默认靠近源文件

单元测试和组件测试默认与被测文件同目录，命名为：

```text
[源文件名].test.ts
```

示例：

```text
src/components/ScaleLayout.vue
src/components/ScaleLayout.test.ts

src/store/auth.ts
src/store/auth.test.ts

src/api/auth.ts
src/api/auth.test.ts
```

### 跨模块测试放 `tests/`

当测试需要同时覆盖多个模块，或更像用户流程而不是单个文件行为时，放到 `tests/`：

```text
tests/integration/login-flow.test.ts
tests/integration/router-auth.test.ts
tests/e2e/login.spec.ts
```

当前项目已有 `tests/setup.ts`，用于测试环境初始化和公共 setup。不要把业务测试文件直接放进 `tests/` 根目录，跨模块测试应放入 `tests/integration/` 或 `tests/e2e/`。

### E2E 配置

当前项目已使用 Playwright 作为 E2E 测试工具，默认只配置 Chromium，适合本项目 PC 后台管理场景。

判断需要新增 E2E 测试时，必须先询问用户，并说明为什么单元、组件或集成测试不足以覆盖该风险；用户确认后再新增 `tests/e2e/*.spec.ts`。

相关约定：

- `playwright.config.ts`：E2E 配置，`testDir` 指向 `tests/e2e`，并通过 `webServer` 自动启动本地开发服务。
- `tests/e2e/`：E2E 测试文件目录，文件名使用 `*.spec.ts`。
- E2E 运行命令以 `package.json` 的 `scripts` 为准，避免文档和脚本不同步。
- Vitest 已排除 `tests/e2e/**`，避免单元/组件测试误扫描 E2E 用例。
- 如果本机首次运行 Playwright 且缺少浏览器，按 Playwright 输出提示安装浏览器。

## Mock 与测试数据规则

- 只 mock 外部边界：网络请求、浏览器 API、时间、随机数、第三方不可控能力。
- 不要断言 mock 组件本身是否存在；应断言被测代码产生的真实行为。
- mock API 响应时，字段结构要尽量贴近真实响应，避免只写当前断言用到的字段。
- 如果 mock setup 比测试主体还复杂，优先考虑调整代码边界或改写为集成测试。
- 不为了测试向生产代码添加只在测试中使用的方法或分支。

## 浏览器运行时验证

涉及页面、组件、样式、交互、接口联调或性能问题时，应使用当前环境可用的浏览器插件或 MCP 浏览器工具做运行时验证；不绑定具体工具名。自动化测试不能替代浏览器运行时验证。

浏览器运行时验证适合发现视觉布局、真实交互、Console、Network 和接口联调表现问题，但它不是可提交、可复跑、可进 CI 的测试资产。能自动化沉淀的业务行为，应补到对应的单元测试、组件测试或集成测试；只有真实跨页面核心链路风险无法由前述测试层覆盖时，才考虑新增 E2E，并按 E2E 规则先询问用户。

验证记录至少说明：

- 页面路径。
- 核心操作。
- Console 是否有错误。
- Network 是否发出预期请求，状态是否符合预期。
- 布局是否存在遮挡、溢出、错位。

纯工具函数、纯类型、纯文档或无 UI 运行时影响的改动，可以不做浏览器验证，但需要说明原因。

## 实施计划中的测试策略模板

计划中可以使用以下结构：

```markdown
## Test Strategy

- 新增/修改测试文件：
  - `src/api/auth.test.ts`：验证接口路径、请求参数和不提交明文密码。
  - `src/store/auth.test.ts`：验证 token 初始化、保存、清理和登录态。
- 不写自动化测试的部分：
  - 登录页视觉布局通过浏览器运行时验证检查，不单独写快照测试。
- 验证命令：
  - `pnpm test -- --run src/api/auth.test.ts src/store/auth.test.ts`
  - `pnpm test -- --run`
  - `pnpm exec eslint "src/**/*.{vue,ts,js}"`
  - `pnpm exec stylelint "src/**/*.{vue,html,css,less,scss,sass}"`
- 浏览器运行时验证：
  - 打开 `/login`，检查表单校验、验证码刷新、失败提示、成功跳转和 Console/Network。
  - 如果涉及核心页面闭环，说明是否需要 E2E；需要新增 E2E 时先询问用户。
```

## 与需求文档的关系

`docs/requirements/*.md` 只维护需求和实现事实，不维护测试或验收章节。测试策略、测试文件位置和验证流程统一以本文件和 `AGENTS.md` 为准。
